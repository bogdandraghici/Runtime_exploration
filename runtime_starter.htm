Runtime Tab Anatomy

How the main concepts in the Runtime Tab of a FlowX project interact — what each holds, what UI surfaces exist for it, and where they overlap. Written as reference for a complete redesign of the section.

2026-05-14 · scope: runtime/* nav within an application

§ 1System map

The Runtime Tab is the read side of a running FlowX app. Five concepts carry almost all the information shown there. Everything orbits around Builds — the published artifact — but the user-facing entry points are Process Instances and UI Flow Sessions. The arrows below are causal: who selects what, who spawns what, who observes what.

How to read it. A runtime caller (end-user or API) hits the application and the Active Policy resolves them to a specific Build. That Build determines which process / UI-flow / workflow / AI-node definitions are available. The caller's interaction surfaces as either a Process Instance (engine-driven) or a UI Flow Session (user-facing). Either side can trigger Workflow Instances (integration / AI runs), which Live Evals observes for quality and cost. Live Evals also anchors its time-series to Builds for before/after comparison.

§ 2Interaction matrix

Read across: how the row concept relates to the column concept. The diagonal is shaded — a concept's relationship with itself isn't interesting here.

Builds Active Policy Process Instances UI Flow Sessions Live Evals
Builds — A Build is the artifact an Active Policy can point at (either by branch tip or by fixed id). Each Process Instance was started against exactly one Build (buildApplicationVersionId) and reads its definitions from it. Each UI Flow Session was started against exactly one Build (buildAppVersionId). Live Evals time-series are bucketed by Build; deploy markers come from BuildSummary records.
Active Policy Holds policyBranchId and/or policyBuildId — the chosen Build target. — Resolves before a new process starts. Doesn't retroactively change instances already running. Resolves before a new UI flow session starts. Doesn't retroactively change running sessions. No direct relation today. Live Evals reads runs after the fact; the policy decided which Build they ran against.
Process Instances Pinned to a Build at start (buildStatusAtStart + buildApplicationVersionId). Has no override mechanism of its own — inherits whatever policy was effective at start. — Can be spawned by a UI Flow Session. The session lists them in its Processes tab. Each instance shows AI workflow runs in its Workflows tab — these are the same runs Live Evals observes.
UI Flow Sessions Pinned to a Build at start (buildAppVersionId). Same as Process Instances — inherits effective policy at start. Spawns them. The Processes tab on the session lists every spawned instance. — Each session shows AI workflow runs in its Workflows tab — same runs as Live Evals.
Live Evals Buckets all metrics per Build; supports left-vs-right Build comparison. None today. Indirectly: workflow runs triggered by PROCESS_ENGINE belong to a Process Instance. Indirectly: workflow runs triggered by CHAT / UI_FLOW belong to a UI Flow Session. —

Named data flows

  1. How a Process Instance picks its Build. The runtime engine asks the build-mgmt service for the effective policy for the calling user (using their username + roles). That returns either a fixed build id or a branch id (the branch's latest build is then used). The resolved id is stamped onto the instance at start as buildApplicationVersionId and never changes.
  2. How a UI Flow Session spawns Process Instances. While the session is running, the UI flow can trigger sub-processes (engine starts them in the same Build context). Those instances are listed under the session's Processes tab and remain navigable individually.
  3. How a workflow run reaches Live Evals. Workflows can be triggered by PROCESS_ENGINE (from inside a Process Instance), by CHAT (from a UI flow chat node), or by UI_FLOW (from a UI flow action). Every run, regardless of trigger, is emitted to the telemetry / scoring pipeline. Live Evals reads those runs and aggregates by workflow, AI node, Build, observation point values, and time window.
  4. How a Build comparison works in Live Evals. The user picks two BuildSummary ids on the comparison page. Live Evals computes per-scorer and per-AI-node deltas between the two periods, plus decision-distribution shifts per observation point. Builds with no shared scorers return an "incompatible" result.
  5. How an alert reaches the UI. A LiveScorerBinding has a threshold. When the rolling score crosses it, a LiveAlertState is emitted with status: 'breached' and a breachedSince timestamp. The Live Evals landing page renders these as an inline orange warning inside the workflow table card, locating the breach as {ScorerName} on {WorkflowName} / {NodeName}.

§ 3Builds connective tissue

A Build is an immutable, published snapshot of an application. It packages every definition the runtime needs — process definitions, UI flows, workflows, AI nodes, configuration parameters. Every other concept in the runtime tab either selects a Build, executes against one, or annotates by one. Builds is therefore the implicit anchor of the entire section.

Identity

buildId
string
Globally unique. Pinned by Active Policy or by an already-running instance.
branchId / branchName
string
Which branch the build came from. Policy can track latest on branch instead of a fixed id.
buildTag
string
Human label (e.g. v1.4). Shown on chips and time-series markers.
buildDate
date
When it was cut. Drives ordering and Live Evals "deploy marker" overlay.
isActive
boolean
Whether it is currently in use by some policy resolution. Surfaces as a "broadcast" icon on Build chips.

UI surface

  • Builds list (runtime/builds-list) — the default landing page for the runtime tab. Lists builds per branch with metadata, status, and a child route at workflow-instances for browsing the workflow runs that happened against a given build.

Connection points

  • Active Policy stores either a policyBranchId or a policyBuildId referencing this entity.
  • Process Instance and UI Flow Session each carry a buildApplicationVersionId / buildAppVersionId stamped at start.
  • Live Evals exposes Builds as time-series markers, as the comparison axis, and as the "active build" badge in workflow / node detail pages.
Asymmetry to note. Build is the only concept everything else points at, but the Builds page itself has no view of "what's currently deployed where". The "where" lives in Active Policy + overrides. Reconstructing the picture requires visiting both pages.

§ 4Active Policy

Active Policy decides which Build a runtime caller is served. It has a per-application default plus optional per-username / per-role overrides resolved by priority. The resolver — "given this user with these roles, which Build will they get?" — is exposed in the UI as a test-effective dialog.

Policy types

  • LATEST_ON_BRANCH — track a branch's tip. The build resolves to the most recent build cut from policyBranchId.
  • FIXED_VERSION — pin to a specific policyBuildId. Does not advance even if newer builds appear on the branch.

Effective policy resolver

POST /test-effective takes { username, roles[] } and returns the resolved policy together with a resolutionPath — an ordered list of { description, matched } entries showing which override rules were checked and which one won. This is the only place the user can see WHY a particular user is getting a particular build.

UI surface

  • Main view (runtime/active-policy): default-policy form with branch + build pickers and policy-type radio.
  • Overrides sub-view: table of overrides with create / edit / enable-toggle / delete + priority editing.
  • Test-effective dialog: try a username + roles set, see the resolved policy and resolution path.

Connection points

  • → Builds. Policy points at a Build (fixed) or a branch (latest). Re-pointing the policy doesn't migrate already-running instances.
  • → Process Instances / UI Flow Sessions. Resolved before creation; effective for new instances only. Already-running ones keep their original build.
  • ↔ Config Param Overrides. Structurally parallel feature (default + per-target overrides + resolver) on a different page; see § 9.

§ 5Process Instances

A Process Instance is one BPMN-style execution of a process definition. It has state (running, on hold, finished, errored, aborted, …), a set of tokens sitting on currently-active nodes, variables (paramValues), and a tree of sub-processes. The detail panel exposes these through eight tabs, each its own data surface.

List page

  • Columns: UUID, Definition name, Exceptions (count badge), Status, Current Node Name, Start date.
  • Filters: definitionName (definition picker) and keyValue (variable-value search).
  • Row actions: open details, plus edit-variables when the user has wks_process_instance_variables_edit.
  • Routes: runtime/active-instances (list), runtime/active-instances/:uuid (detail).

Connection points

  • ← Active Policy. Pinned to a Build at start; never re-pointed.
  • ← UI Flow Sessions. A UI Flow Session can spawn Process Instances. The session-detail endpoint returns both the session and its spawned process instances bundled together — the linkage is resolved server-side, not exposed as a foreign key on ProcessInstance.
  • ↔ Failed Process Start / Failed Triggers / Exceptions. Three sibling pages that surface processes that didn't behave normally (didn't start, were triggered but didn't start, or threw at runtime).
  • → Workflow Instances. The Workflows tab on the detail panel filters workflow runs to those with trigger type PROCESS_ENGINE.
  • → Live Evals. Those same workflow runs feed Live Evals aggregations; the workflow detail page in Live Evals shows them grouped by workflow + Build.

§ 6UI Flow Sessions

A UI Flow Session is one user-facing execution of a UI flow — chat, conversational UI, multi-step form, embedded experience. It's lighter than a Process Instance: status, variables, audit, and the bridge into the process layer (it spawns Process Instances). It is the only first-class runtime entity that represents user-facing behavior directly.

  • Columns: ID, UI Flow, Status, Start Date.
  • Filters: definitionName (UI flow picker) and keyValue (variable-value search).
  • Routes: runtime/ui-flows (list), runtime/ui-flows/:uuid (detail).
  • Permission gate: ui_flow_session_read (application-scoped).

Connection points

  • ← Active Policy. Same pin-at-start behavior as Process Instances.
  • → Process Instances. The Processes tab is populated by the session-detail API, which returns the session and its spawned process instances together. The relationship is server-resolved; there is no direct foreign key on ProcessInstance pointing back at the session.
  • → Workflow Instances. The Workflows tab filters workflow runs to triggers CHAT and UI_FLOW.
  • → Live Evals. Those same runs feed Live Evals — the same workflow surfaces in three places (here, in the parent Process Instance, and in Live Evals).
Structural symmetry with Process Instances. Both have the same anatomy: list page, detail panel with tabs, Build pinned at start, variables, audit log, Workflows tab. They diverge on what they emphasise — UI Flow Sessions emphasise spawned processes, Process Instances emphasise tokens and exceptions. This symmetry is the strongest redesign hook in the runtime tab; see § 9.

§ 7Live Evals

Live Evals is the runtime observability surface for AI workflow quality. It sits next to Builds in the nav and is application-scoped (cross-workflow). It exposes three tiers per workflow — Telemetry (always-on), Workflow scorers (opt-in), AI nodes — under two orthogonal filters: Time window and a Decisions facet driven by designer-marked observation points.

Three-tier workflow page

  1. Tier 1 — Telemetry. Free of charge, always on. Terminal-outcome distribution, path-class top 5, mean path length, retry pressure, fallback-model usage, p95 e2e latency, total cost, instance count.
  2. Tier 2 — Workflow scorers. Opt-in. Time-series of bound scorers with Build deploy markers + aggregate scorer tiles + recent workflow instances list.
  3. Tier 3 — AI nodes. One card per AI node in the workflow with bound-scorer chips and an open-alert badge; click drills into the node detail.

Routes

  • runtime/live-evals — landing (workflow list with summary scores + alert banner inside the table card)
  • runtime/live-evals/:workflowId — workflow detail (three-tier)
  • runtime/live-evals/:workflowId/settings — bindings + observation-points tabs
  • runtime/live-evals/:workflowId/nodes/:nodeId — AI-node detail
  • runtime/live-evals/:workflowId/builds/:leftBuildId/compare/:rightBuildId — Build comparison

Connection points

  • ← Workflow runs from Process Instances and UI Flow Sessions. Live Evals reads the same runs that surface in the Workflows tab of both Process Instance and UI Flow Session detail pages.
  • ↔ Builds. Build markers overlay every time-series; comparison is build-anchored; the workflow card's "active build" badge reads the same BuildSummary.
  • ↔ Active Policy. No direct relation today. The policy decides which Build runs, Live Evals reports what happened on each Build.
Permission piggyback. The route currently uses wks_builds_read; wks_live_evals_read is reserved in PermissionValue but the backend doesn't grant it yet. A real launch needs to flip this.

§ 8Surrounding runtime modules

The runtime tab has more entries than the four focal ones. Each below interacts with the core concepts; treat this as a quick map, not full documentation.

Builds list

The default landing for the whole runtime tab (runtime/builds-list). Lists every published build, grouped by branch. Has a workflow-instances child outlet for inspecting workflow runs scoped to a specific build. Links to: Active Policy (which build the policy points at), every running instance (which build they were started on), Live Evals (per-build time-series).

Scheduled Processes

Time-based triggers that auto-start Process Instances on a schedule (runtime/scheduled-processes, gated by wks_active_policy_read). Links to: Process Instances (the things it starts), Failed Process Start (when starts go wrong), Failed Triggers (when the schedule fires but the start fails).

Manage Triggers

Event / external-trigger configuration that starts Process Instances on external signals (runtime/manage-triggers). Links to: Process Instances (the things it starts), the same Failed Triggers / Failed Process Start surfaces.

Failed Process Start

Cases where the engine refused to start a process (bad input, missing build, validation, …). Sibling to Process Instances in the Runtime Processes nav group. Links to: the Process Instance that would have been created, and the trigger / scheduled process that asked.

Failed Triggers

Triggers that fired but didn't successfully reach a process start. Drill-down at runtime/failed-triggers/:id. Links to: Manage Triggers (configuration), Failed Process Start (downstream outcome).

Exceptions / Exception details

Incidents on running Process Instances (runtime/exceptions, runtime/exceptions/:id). The same data surfaces inside the Process Instance Exceptions tab; this list is a workspace-wide cross-process view. Links to: Process Instances (the host instance for each incident).

Task Manager

The user-task inbox surface (runtime/task-manager, gated by wks_tasks_read). Tasks are created by Process Instances that reach a human-task node. Links to: Process Instances (the source of each task).

Config Param Overrides

Per-runtime-user / per-role overrides of configuration parameters (runtime/overrides, gated by wks_config_param_overrides_read). Structurally similar to Active Policy overrides but for arbitrary config params instead of build version. Links to: Active Policy (conceptual mirror), every running instance (which params it sees).

§ 9Friction / redesign hooks

Observations, not recommendations. Each is a place where the current shape of the runtime tab makes a question harder to answer than it should be. They're meant to anchor design conversations, not pre-empt them.

Active Policy and Config Param Overrides have the same shape duplication

Both are "default + per-username/role overrides with priority + effective-resolver". They live on separate pages with different table chrome, different override-form modals, and different test-effective UX — but the underlying conceptual model is identical. A redesign could collapse them into a single "Runtime configuration" surface where Build-version is just one of several configurable axes.

Process Instances and UI Flow Sessions are near-symmetric parallel concepts

They share: list page, detail bottom-panel, Build pinned at start, variables, audit log, Workflows tab. They diverge on: tokens vs status, exceptions vs spawned processes. They live under different top-level nav entries (Runtime Processes > Process Instances vs UI Flow Sessions at top level). A user looking for "a thing that ran in production five minutes ago" has to already know which kind it was.

AI workflow runs appear in three places duplication

The same workflow run shows up in (a) the parent Process Instance's Workflows tab, (b) the parent UI Flow Session's Workflows tab if any, and (c) Live Evals. Each view applies a different lens — engine context, user context, quality aggregations — but the underlying entity is the same workflow execution. There's no way today to navigate from a single workflow run to its sibling lenses.

No unified "what's running right now" entry point missing surface

Process Instances and UI Flow Sessions are the only sources of live state, but a user must already know which kind of artifact they're looking for to choose a page. There's no cross-cutting "live runtime" view that lists both, no global activity feed, and no single place to scan for anomalies across both surfaces.

Three separate "things went wrong" surfaces fragmentation

Failed Process Start, Failed Triggers, and Exceptions are three sibling pages with the same emotional purpose ("show me what's broken"). They differ on when in the lifecycle the failure happened (before start, at trigger, mid-execution). A redesign could consolidate them into a single "incidents" view filtered by failure stage, with deep-links into the originating Process Instance or trigger.

Builds is the implicit centre, but its page has no deployment view missing surface

Every other runtime concept references a Build. But the Builds list itself shows builds-as-artifacts, not builds-as-deployments. Answering "what build is the policy actually serving right now, and to whom?" requires opening Active Policy, then opening test-effective with sample user inputs. A redesign could move the deployment lens onto the Builds page as a first-class column ("currently effective for: 87% of users / role X / username Y").

Status models are inconsistent data shape

Process Instances have a strongly-typed ProcessState enum with ten well-defined values. UI Flow Sessions have a free-form status: string. Live Evals has yet another set (workflow completion status, alert status). A redesign that surfaces multiple kinds of running things together will need a status-domain decision — converge, normalise, or render distinctly.

The detail panel pattern is repeated, slightly differently, three times component drift

Process Instance details, UI Flow Session details, and Live Evals invocation / instance modals all use a bottom-panel-with-tabs metaphor, but each is its own component with slightly different tab chrome, slightly different metadata header, and different state-pill treatment. The recent Live Evals work aligned its modal with the agent-builder report-case modal style; the runtime detail panels haven't been touched. A redesign could reach a shared "runtime entity detail panel".

The Workflows tab on instances is a redundant Live Evals slice latent integration

flx-instance-workflows lists workflow runs filtered by trigger type. Live Evals can already render that exact list (with richer scoring). A redesign could replace the tab content with an embedded Live Evals fragment — same data, one source of truth for AI runs — or invert the relationship and let the Workflows tab deep-link out to Live Evals for any scored run.
