§ 2Redesign philosophy

Three personas open this tab for three different reasons — what's broken?, how's my workflow behaving?, what did the system do today? The redesign serves all three by reorganising what already exists, not by inventing anything new.

01 · ORIENTATION

Activity-first, not artifact-first

The tab is organised around verb-questions, not entity-kinds. Users follow their intent (what's broken? what's running?) instead of guessing which catalogue holds the answer.

02 · CONSOLIDATION

Merge near-symmetric concepts

When two pages share the same grammar, they become one page with a column or a tab. Four such merges collapse parallel mental models into single surfaces.

03 · CANONICAL HOME

One source of truth

Each concept lives in one place. Other surfaces show filtered slices — windows into the same canonical implementation, never parallel ones.

04 · LEGIBILITY

Self-teaching surface

Every concept is definable from inside the product. Help icons, explanatory empty states, and tooltip-revealed raw vocabulary mean any user can learn the system by using it.

How each page or entity in the current Runtime tab moves to the redesign.

UI Flow Sessions
Two separate top-level entries. Near-symmetric anatomy (list + detail with tabs), but with different chrome, different status models, and live under different parents in the nav. Starter spec called for 8 instance-detail tabs (Variables · Tokens · Subprocesses · Exceptions · Workflows · Trigger · Notifications · Audit), each thin and easy to miss.
 → Merged
Instances
One list, type as a column (green P · blue F). Five-state normalized status pill (Running / Waiting / Completed / Failed / Terminated). Shared detail panel with "Activity tree" tab that adapts to type. Cross-link via "spawned by" indicator.

6-tab detail panel. The starter's 8 tabs collapse to 6 (Overview · Variables · Activity tree · AI runs · Incidents · Audit) without dropping content. Dropped tabs fold into existing ones:
  • Subprocesses render under Activity tree as a recursive tree of child instances (cycle-guarded). Each row is click-through to the child's detail.
  • Trigger initiator (manual · cron · webhook · process · uiflow) shows as a KV row in Overview with a kind glyph and a linkable trigger or parent reference.
  • Audit timeline rebuilt as a full chronological log weaving six event sources: trigger fire (from initiator.when), instance start, per-token nodesActionStates (token-id chip + verb mapping: enter→arrived, complete→finished, execute→executing, abort→aborted), back-seq flags as ↺ Back to <node>, subprocess spawns anchored to each child's own startedAt, terminated lifecycle (previously only failed/completed surfaced), and notifications (envelope dot for sent, red bang for failed). New .dot.node glyph (small outlined circle) distinguishes node-action rows. A typical instance now renders ~7 events instead of ~2.
  • Tokens in Activity tree are expandable — chevron reveals raw ProcessState pill (with description tooltip), per-node action history table, and a back-seq chip when applicable.
  • Variables gain a per-row pencil affordance opening a centered .app-modal (Esc + scrim close, focus restored to opener). Edits live in STATE.variableEdits in-session; an "edited" badge marks overrides; "Revert to original" undoes per-key.

Active Policy
Config Param Overrides
Two pages, identical underlying shape (default + per-user/role overrides with priority), separate UIs, two test-effective dialogs.
 → Merged
Configuration
Two axes (Build version + Config parameters) on one page, shared override grammar. Persistent right-side "Test as user" resolver shows BOTH outcomes in one shot with full resolution path.

Working override CRUD. Edit and Delete on each row open the shared .app-modal scaffold (now driven by a STATE.activeModal dispatcher so it can host any client). Build axis exposes target radios, policy-type radios (Latest-on-branch ↔ Fixed-version with dimmed inactive dropdown), branch + build selectors, and priority. Param axis uses a key=value textarea with the defaults-keys listed below as mono chips. Saving mutates the in-session policy and bumps the audit fields. Delete is a confirm modal with a target/policy summary and a filled red primary button. The resolver auto-re-runs if a result was already on screen.

Audit metadata (createdBy/Date, modifiedBy/Date) added to every override and surfaced two ways: a row title tooltip and a dedicated footer block inside the Edit modal.

Param-axis resolution path. The resolver now emits an ordered paramPath parallel to the build axis's path. Each step is classified — does-not-match · disabled (with the keys it would have set) · matches-and-wins · matches-but-superseded (every key already won by a higher-priority rule) · partial-win (sets some keys, others already won). Terminal step calls out which keys fell through to defaults. Same .path-step renderer as the build axis.

One page, four tabs (Scheduled · External · Recent fires · Failed fires). Trigger-stage failures double-surface in Incidents. Two-word label preserves both mental models — users hunting for "the nightly batch job" find it under Schedules, users wiring a webhook find it under Triggers.
Trigger detail (none)
Clicking a trigger row in the old Manage Triggers / Scheduled Processes pages either did nothing or revealed thin config-only modals. No fire history alongside config. No path to incidents.
 → New
Trigger detail panel
Right-slide panel with three tabs — Configuration (type · schedule · spawns · status · 24h error count · audit fields), Recent fires (chronological history per trigger with spawned-instance links), and Failures (conditional; only when the trigger has any failed fires; rows deep-link to the corresponding Incident). Reachable from the Triggers list, from Incident detail's source-entity link when sourceEntity.kind === 'trigger', and from Instance Overview's Trigger initiator row.
Builds list
Lists builds as artifacts. Has no deployment view — "what is currently serving whom" requires bouncing to Active Policy + test-effective. No workflow-runs-scoped-to-build view.
 → Enhanced
Builds
Same list, new coverage column per row (% traffic, # users). Detail panel adds Deployment tab (who is and isn't being served), a Workflow runs tab (every AI run on that build; count badge turns red if any run has a breached scorer), and a Comparison tab that lists workflows running on this build and links into the workflow-scoped comparison page. The "Compare to…" header button auto-routes to the most-relevant workflow's comparison page using this build as one side.
Live Evals
Three-tier flat page with telemetry, scorers, AI nodes. Workflow chosen via a chip selector that drives the whole page. Lives in isolation — workflow runs are duplicated in instance detail tabs. No way to navigate from a single AI run to its sibling lenses (parent instance, build, Live Evals). No configuration surface — bindings and observation points existed in spec but had no UI.
 → Enhanced
Live Evals — landing + per-workflow drill-in
IA restructure. Workflow promoted to a URL segment. #/live-evals is now a cross-workflow landing grid (one card per workflow with stats, scorer chips, breach badges). #/live-evals/wf_xxx is a workflow detail page with breadcrumb and tabs [Overview] [Settings]. #/live-evals/runs is the cross-workflow runs list. No more chip-selector ambiguity — each workflow's bindings and observation points are scoped to their own URL.

Overview tab — page-level time window (1h / 24h / 7d / Custom) and a Decisions facet (one chip per observation point, cycles through unique values). Tier 1 now shows 8 metrics plus terminal-outcome distribution and top-5 paths. Tier 2 renders inline SVG line charts per scorer with dashed-orange threshold and vertical build-deploy markers. Tier 3 cards drill into an AI node detail panel (Trace / Invocations / Scorers). Recent runs section below.

Settings tab — Scorer bindings table (scope, sample rate, daily budget, threshold, enabled toggle) and Observation points table (dimension, category, bound node, unique values, enabled toggle). Disabling an observation point removes its chip from the Overview filter row.

Build comparison sub-route at #/live-evals/wf_xxx/builds/<l>/compare/<r>. Build picker header (swappable left/right) followed by aggregate metric deltas, per-scorer deltas with regression flags, per-AI-node latency/cost shifts, and decision-distribution shifts per observation point. Archived builds (no scorer history) trigger an incompatible empty state.

WorkflowRun detail panel (Trace / Variables / Scores) reachable from every surface; parent chip jumps back to the spawning Process Instance or UI Flow Session — the cross-lens connector that didn't exist before.
Task Manager
User-task inbox. Tasks list their originating process implicitly. No reverse linkage. No detail panel — clicking a task did nothing useful.
 → Enhanced
Tasks
Same list. Each row carries originating Instance ID with click-through; same task surfaces in the Instance's Activity tree tab. Fully cross-linked.

Detail panel. Three tabs — Overview (source link + assignee + due + priority + Claim / Open form / Reassign actions), Form (synthesized fields per task type — loan tasks get Decision + Amount approved + Notes; KYC tasks get Identity verified + Notes; etc.; disabled when completed or not assigned to you), History (created / claimed / completed timeline). Working in-session Complete that moves the row out of Pending into Completed.
